## 1. 结论先行

1. 工资表本质是一台「取数 + 公式 + 人工修正」的机器：底薪来自职级表、考勤来自钉钉导出、社保公积金来自代缴机构 Excel、绩效来自规则计算、少数列纯手工。目前全流程人工，**3 月表已发现 9 处以上人为错误（含一处资金风险级）**（§2.5），自动化价值明确。
2. 模块定位：新领域模块 `app/salary/`，做「月度工资批次」工作流——同步钉钉考勤 → 导入社保公积金 → 引擎计算 → 人工复核修正 → 锁定 → 导出与人工表同构的 Excel。**业务提成不进本模块**（现有提成域按订单回款独立发放，3 月工资表本身也不含提成列）。
3. 考勤通过钉钉开放接口拉取（平台 `app/dingtalk/client.py` 已有 token 基建，补考勤 API 封装即可），全勤奖、缺勤扣款、年假备注全部自动生成。
4. 规则全部参数化 + 版本化（职级表注明「2026/04/01-更新新版本为止」，规则本身就是带生效期的）。
5. 分三期：P1 核心复算引擎 + 导入导出；P2 绩效自动化 + 钉钉工资条；P3 晋升调级联动 + 个税。三期边界见 §8。

## 2. 材料分析总结

### 2.1 《2026年3月工资表.xls》（核心输出物，两张 sheet）

**Sheet2「工资表（明细）」是计算真相源**，23 列：

| 列 | 性质 | 来源 |
|----|------|------|
| 序号/入职时间/工号/姓名/部门/职位 | 固定信息 | 员工薪资档案 |
| 应出天数 | 规则 | 满月=31（当月自然日）；月中入职/实出<15 天=当月工作日数（3 月为 22） |
| 实出天数 | 规则 | 应出 −（事假时长/7.83）−（病假时长/7.83×折算比例），考勤来自钉钉 |
| 应发工资（底薪） | 规则+档案 | 职级表（业务 P/M、跟单 F/M）或手动定薪；月中调薪/转正按天加权（陈佳乐 3775 介于试用 3500 与转正 4000 之间，但按 31 天基数复算为 3774.19 对不上——精确加权口径待 HR 确认，见 §9.2） |
| 奖励 | 规则+手动 | 单项奖金（月度销售额阶梯，3 月全空） |
| 绩效 | 规则+手动 | 业务员/管理岗按绩效规则算，社媒运营另有规则（材料缺），个别岗位手动 |
| 工龄 | 规则 | 200 元/满一年，上限 2000；按工资月所在月份过周年即上调 |
| 全勤奖 | 规则 | 100 元；无事假、病假≤8h、无迟到/早退/漏打卡/旷工；**年假不影响全勤** |
| 小计（增项） | 公式 | 奖励+绩效+工龄+全勤 |
| 社保/公积金 | 导入 | 与当月社保、公积金明细「个人部分」逐人匹配 |
| 缺勤扣款 | 公式 | −底薪/应出天数×缺勤天数（已验证：谷振尧 10000/31×4=1290.32；王槐竹 6000/22×5=1363.64；李晓雨 5000/21.75×8.18=1880.46） |
| 其他款 | 手动 | 正负皆可（如刘德明 +1000） |
| 小计（减项） | 公式 | 社保+公积金+缺勤+其他款 |
| 补贴项 | 手动+规则 | 每月手动；**保底工资补足自动化**：实发 = max(计算实发, 保底 − 缺勤扣款)，差额进补贴项（已验证：张甜甜 699、隋晓茹 1470、凯丽比努尔 1488、刘也 1679 复算吻合；负例也自洽——徐瑞萍/张紫娟保底 2026/04 起生效，3 月未补） |
| 实发工资 | 公式 | round(底薪 + 增项小计 − 减项小计 + 补贴项)，四舍五入到元 |
| 个税/扣除个税实发 | 手动 | 3 月全员为 0（申报走税务系统，表内仅留位） |

**Sheet1「工资汇总表」在金额上是明细的投影**（发放/公示口径）：按大部门分组（后综部 39 人 / 业务部 27 人，与明细的细部门是多对一映射），列 = 序号/部门/姓名/职务/入职日期/中行银行卡号/请假时间/实发工资/备注。注意：**职务列两表大面积不一致**（明细 vs 汇总：刘琳琳 M6/M2、毕晓珍 M6/M5、夏新月 M3/M1、宋皓月 M5/业务、李宝珠 M6/INS开发、田雯 M4/业务、刘也 外贸业务员/社媒运营）——汇总表职务疑似另一套对外/滞后口径，真相源归属见 §9.5。其余：
- 「请假时间」列 = 自动文案：`本月年假X天，本年度剩余年假Y天`（试用期员工则写底薪约定）→ 需要**年假台账**；
- 「备注」列 = 自动文案：`扣社保XX元，公积金XX元。`；

### 2.5 人工流程的错误实证（自动化依据）

3 月表交叉核对发现 **9 处以上**真实错误，全部属于「复制粘贴/誊抄/错行」类，系统化后此类错误结构性消失：

3. **身份证号复制错误**：公积金表中刘也的身份证号与姜婷相同（社保表中可见刘也真实证号，两处不一致）；
4. **工号重复**：刘德明/江倩倩同为 293，张桂云/陈佳乐同为 319；另有吕德洋 3 与刘美美 003 归一化后撞号；

## 3. 模块边界与总体架构

### 3.1 边界（明确不做什么）

- **不做业务提成**：提成按订单回款发放，归现有提成域；工资表 23 列中无提成列，维持现状分轨；
- **不做个税计算引擎**（P1）：表内个税列保留手动录入，默认 0，申报仍走自然人电子税务局（P3 再评估累计预扣法，见 §8）；
- **不做晋升自动判定**（P1）：职级是档案字段 + 调级记录，调级动作人工（P3 联动业绩数据自动预警，见 §8）；
- **不改钉钉考勤规则本身**：钉钉是考勤唯一真相源，本模块只读。

## 5. 钉钉考勤集成设计

平台 `app/dingtalk/client.py` 已有 access_token 与通用 get/post，本模块在 `attendance_service.py` 内新增封装（不改 dingtalk 域，只调用其 client——与 approval.py 同层的用法）。

### 5.1 取数路径（三路互补）

| 数据 | 钉钉接口 | 用途 |
|------|----------|------|
| 月度考勤报表列值 | `topapi/attendance/getattcolumns` + `topapi/attendance/getcolumnval` | 与 HR 现在手工导出的「钉钉月度考勤报表」同源，直接拿出勤天数、事假/病假/年假时长等聚合列，是主路径 |
| 打卡结果 | `attendance/list`（≤50 人/批，7 天窗口分页） | 迟到/早退/漏打卡/旷工明细次数 → 全勤奖判定 |
| 请假明细/额度 | `topapi/attendance/getleavestatus`（20 人/批）+ `vacation/type/list` + `vacation/quota/list` | 假期按类型拆分（事假/病假/年假）、年假余额 → 汇总表「请假时间」文案 |

员工映射：档案存 `dingtalk_userid`，初始化用已有 `get_userid_by_mobile(mobile)` 批量回填，匹配不上的进异常清单人工绑定。

### 5.2 折算与判定（引擎输入）

- 事假折天 = 事假小时 / 7.83；病假折天 = 病假小时 / 7.83 ×（1 − 病假计薪比例）——规则原文「相对于的比例」有歧义（×30% 扣减 vs ×70%），3 月无病假样本可验证，**方向待确认（§9.3），P1 按扣减 30% 实现 + 支持手动改**；
- 全勤 = 无事假 且 病假 ≤8h 且 迟到/早退/漏打卡/旷工均为 0。**「年假不破全勤」是从数据反推的**（王京花年假 5.5 天仍得全勤 100），规则图该句文字残缺，待 HR 确认（§9.3）；反向佐证：3 月有 5 人 31/31 全勤却为 0，只能用钉钉迟到/漏打卡明细解释，说明打卡明细路径必须接；
- 应出天数 = 满月员工取当月基准天数（3 月=31，参数 full_month_days，2 月口径待确认 §9.1）；当月入/离职或实出 <15 天 → 取当月工作日数（batch 上存 workday_count）；
- 同步为**拉取式**：批次页点「同步考勤」触发（也支持每月 1 日 scheduler 预拉上月），拉完先落 `salary_attendance` 供人工核对，不直接进计算；
- **历史数据依赖**：P1 验收要复算 2026-03，前提是钉钉仍可回拉 3 月考勤明细（钉钉考勤数据留存有时限），实现第一步先验证可回拉范围，拉不到则 3 月考勤用人工表数值代录完成复算。

### 5.3 失败与兜底

接口全部走 savepoint 单人隔离（宪法硬约定 6）：某员工拉取失败不阻塞整批，标记 sync_failed 进异常清单；支持单人重拉、全量重拉、手工录入覆盖（sync_source=手动）。

## 6. 计算引擎

`calc_service.py` 核心为**纯函数**：`calc_record(profile, grade, attendance, insurance, fund, params, manual_overrides) -> RecordDraft`，不碰 session，管钱逻辑 100% 可单测（DoD 硬要求）。

```
底薪        = 档案定薪 or 职级表底薪；月中调薪/转正 → 按 change_log 生效日按天加权
              （加权天数基数 31 vs 30 vs 工作日待确认 §9.2，3 月陈佳乐 3775 两种口径差 1 元）
奖励        = 单项奖金（P1 手动；P2 按销售额阶梯规则算，见 §8）
绩效        = P1: 手动录入金额（引擎只给参考=满绩效）
              P2: 满绩效 ×（权重×完成率），权重按转正年限/M 岗自动选
工龄工资    = min(200 × 周年数, 2000)，周年数按「工资月内过入职纪念月即+1」
全勤奖      = 考勤判定 ? 100 : 0
社保/公积金 = 导入表按身份证号匹配的个人部分（匹配不到→0 并进异常清单）
缺勤扣款    = 底薪 / 应出天数 × (应出 − 实出)
补贴项      = 手动 + 保底自动补足：若 计算实发 < 保底 − 缺勤扣款，补差额（生效区间内）
实发        = round(底薪 + 奖励 + 绩效 + 工龄 + 全勤 − 社保 − 公积金 − 缺勤扣款 + 其他款 + 补贴)
              其中「其他款」是带符号净额（可正可负，如刘德明 +1000 使减项小计变为 +283.12），
              社保/公积金/缺勤扣款恒为非负、按减项处理
```

重算语义：任何输入变化（考勤重拉、导入更新、参数修正）→ 重算只更新 auto_value，人工 manual_value 保留并高亮「auto 已变化但被人工覆盖」的行，复核者决定是否放弃覆盖。

**负数实发拦截**：低底薪 + 大额缺勤 + 社保可算出负数（3 月最低已到 1114 元）。final 实发 < 0 的行禁止进入 confirmed，强制进异常清单人工决策（清零挂账下月 / 其他款冲抵 / 照发负数需 admin 显式确认），银行盘生成时二次校验金额 > 0。

## 7. 工作流、页面与权限

### 7.1 批次状态机

```
draft → attendance_synced → imported → calculated → reviewing → confirmed(锁定) → exported/paid
```

- confirmed 前可任意重算；confirmed 时将档案字段（含银行卡号）快照进 salary_record，此后全表只读，修正需 admin 解锁并留痕；
- 复核是多人协作（制表/审核/复核三签），手动列编辑走 row_version 乐观锁：提交时版本不符 → 提示「该行已被他人修改」并展示对方改动，不做静默覆盖；
- 每步完成度在批次页用步骤条呈现，未就绪项（考勤未同步 N 人、社保未匹配 M 人）直接显示在步骤上，反馈引导行动。

### 7.2 页面（navigation.js 各加一条 entry，新分组「薪资管理」）

| 页面 | 内容 | 标杆 |
|------|------|------|
| 工资批次列表 | 月份、状态、总额、操作 | ExpoLeads 编排（useListPage） |
| 批次工作台（核心页） | 步骤条 + 明细表格（23 列，手动列行内编辑）+ 异常面板 + 导入对话框（上传用 `components/AppUpload`）+ 导出按钮 | 大页拆 `composables/useSalaryPeriod.js` |
| 员工薪资档案 | 档案 CRUD、职级/赛道、保底、参保主体、钉钉绑定、调薪记录 | DictManagement 标杆 |
| 规则配置 | 职级薪级表（按版本）、规则参数、大部门映射 | DictManagement 标杆 |

异常面板是体验核心（不要让用户思考）：未匹配社保、考勤失败、卡号/身份证重复、保底触发、月中调薪加权、auto 被人工覆盖但 auto 已变 —— 全部聚合成待办清单，点击定位到行。

### 7.3 导出（核心输出物）

openpyxl 生成与人工表同构的 .xlsx：Sheet1 汇总表（按大部门分组、组内重排序号、合计/总计行、自动备注文案、制表/审核/复核脚注）+ Sheet2 明细表（23 列 + 合计行）。另出银行代发盘（姓名/卡号/金额 CSV）作为可选导出。**confirmed 批次的导出一律读 salary_record 快照，不回查活档案**。

### 7.4 权限（宪法硬约定 3/8）

- `salary:read` / `salary:write` / `salary:admin` 三档起步（admin = 锁定/解锁/规则配置/明文导出）；
- 薪资是全平台最敏感数据：navigation entry 用 permission 声明，仅 HR 与管理层角色分配；接口层每端点 `Depends(require_permission)`；
- 员工个人工资条自助查询（self 数据范围）属 P2，届时按数据范围权限规范登记 kind=data。
