# 项目资料协作站（PM Hub）功能设计

> 2026-07-17 | 服务于「阿里国际站智能体搭建」陪跑项目的资料管理与任务跟踪站点\
> 状态：设计稿 v2（已过对抗性审查并修订），待亮哥确认

***

## 一、一句话定位

给智能体陪跑项目建一个**独立子域名网站**：内部成员输入用户名即可进入，集中管理顾问要求的 35 项材料（上传/修改/删除全程留痕、版本自动编号、AI 自动生成与上一版的差异概要），并用轻量看板跟踪项目任务，让所有人打开首页就能看清「项目推进到哪了、材料备齐了多少」。

## 二、已确认的方向决策（2026-07-17 与亮哥逐题确认）

| 决策点  | 结论                                                      | 理由                                                      |
| ---- | ------------------------------------------------------- | ------------------------------------------------------- |
| 架构形态 | **后端复用方舟平台**（新领域模块），**前端完全独立应用**（独立设计语言，与方舟 UI 无任何视觉关联） | 新增运维接近零、AI 差异对比直接用平台 ai.service；用户感知上是一个全新的独立网站         |
| 核心建模 | **任务与资料是两个独立对象，可互相关联**；建站时用 35 项材料清单预置初始化               | 清单里有非文件类事项（API 授权、workshop 定标准），且项目会产生清单之外的任务（评审会、顾问对接） |
| 验证方式 | 输入预设用户名即进入，无密码（亮哥明确要求，风险与缓解见第八节）                        | 内部临时使用，够用就好                                             |

**放弃的备选**：钉钉知识库/飞书文档能覆盖版本与评论，但给不了「AI 差异概要」和「按材料清单定制的进度视图」这两个核心价值，且对外部顾问开放企业协作工具有权限管理成本。

## 三、用户与访问模型

### 3.1 进入流程

1. 访问 `pm.leshine.work`（子域名待定，见第十节）→ 极简门牌页：站点名 + 一个输入框
2. 输入用户名 → 后端校验白名单 → 命中则签发轻量 token（HMAC 签名，含用户名 + 有效期 30 天）→ 存 localStorage，30 天内免验证直接进入
3. 未命中 → 统一提示「无法验证，请联系亮哥」，不区分「用户名不存在」（防枚举），连续失败限速
4. **限速取 IP 的口径**：生产链路是云 Nginx → frp → 本地 8002，后端直接看到的 remote addr 是隧道地址（全员同 IP）。限速必须按 `X-Forwarded-For` 首跳取真实 IP——实施时需确认云 Nginx 已设该头、uvicorn 开 `--proxy-headers`；确认前限速仅按用户名维度做（同一用户名每分钟 5 次），不按 IP

### 3.2 身份与留痕

* **没有角色区分**：所有进入者权限相同（都可上传/修改/删除/管理任务）——不做复杂 RBAC，靠留痕约束行为

* **所有写操作记审计日志**：谁（用户名）、何时、做了什么（上传 v3 / 改了任务状态 / 删了资料）、对象是什么

* **全局顶栏常显「当前身份：xxx」+ 一键切换**（清 token 回门牌页）。无密码机制下"谁的浏览器 = 谁的身份"，共用电脑会把操作静默记到错误的人头上——留痕错人比没留痕更糟，必须让身份时刻可见、可一键纠正

* **token 每请求回查白名单**：`require_pm_member` 验签之后还要查 `ark_pm_members` 该用户名是否仍有效——从名单移除某人立即生效，不用等 token 过期，也不用全局吊销拉全员陪葬

* 用户名白名单存数据库表，初始 seed 写入（名单待亮哥提供），后续增删直接改表（人少，不做管理页）

## 四、信息架构

```
pm.leshine.work
├── /            门牌页（用户名验证）
├── /dashboard   总览仪表盘 ← 默认落地页
├── /tasks       任务看板
├── /materials   资料库（清单视图）
├── /materials/:id   资料详情（版本时间线 + AI 差异 + 评论区[P2]）
└── /activity    全站动态（审计日志的友好呈现）
```

共 5 个页面 + 1 个门牌页。刻意压缩页面数量——功能不复杂，质感靠密度和细节。

## 五、Phase 1 功能设计

### 5.1 总览仪表盘（/dashboard）

回答一个问题：「项目现在什么状态？」打开 3 秒内看懂。

* **顶部项目脉搏**：材料完成率（必须项单独突出）、任务完成率、按顾问 Phase 1-4 的分段进度条

* **材料概况块**：按重要级（🔴必须 / 🟡重要 / 🟢锦上添花）分组的状态统计，点击穿透到资料库对应筛选

* **风险提示条**：逾期任务、Phase 1 未齐的必须材料（顾问说了"不齐则核心链路跑不起来"）自动置顶提醒

* **最新动态流**：最近 10 条操作（"Kevin 上传了《价格体系》v3 · AI：调整了 L1+ 客户折扣范围"），每条附 AI 差异的一句话摘要

### 5.2 任务管理（/tasks）

轻量看板，不做甘特图、不做子任务、不做工时。

* **字段**：标题、描述、状态（待办 / 进行中 / 已完成 / 受阻）、负责人（从用户名白名单选）、截止日期、所属 Phase（可选）、关联资料（可多选）

* **视图**：四列看板（按状态）+ 列表视图切换；支持按负责人 / Phase 筛选

* **状态流转交互**：Phase 1 用卡片上的状态徽标点开下拉流转（实现成本低、移动端友好）；看板拖拽留到 Phase 2 打磨项，不算入首期排期

* **受阻状态**必填受阻原因——顾问项目里"等对方给接口权限"这类卡点要显性化

* **关联资料**：任务卡上直接显示关联资料的当前状态徽标，点击跳资料详情

* 所有变更记审计（改状态、改负责人、删除）

### 5.3 资料库（/materials）

清单式主视图，不做网盘式文件夹树——资料集合是有限且已知的（35 项 + 少量新增），清单比树快。

* **组织维度**：按顾问清单的五大分类（产品与报价 / 客户分类与分配 / 知识与内容 / 系统权限与账号 / 历史数据）+ 其他；支持按重要级、Phase、状态筛选

* **资料条目字段**：名称（**同项目内唯一**，避免两条「价格体系」造成下载名与统计混乱）、说明、重要级、所属 Phase、交付类型（**文件 / 线下交付 / 外部链接**）、状态、负责人、当前版本号、最后更新（人 + 时间）

* **状态机**：`未开始 → 准备中 → 已提交 → 顾问确认`，另有「无需提供」终态。状态手动流转（无审批流），流转记审计

* \*\*交付类型「线下交付」\*\*是给账号凭据类材料的（#14-17 阿里/OKKI/物流 API、通知 webhook，以及 #21 海关数据权限、#31 LinkedIn Sales Navigator）：**这类条目禁止上传原文**，上传入口置灰并说明原因，只跟踪状态 + 备注交付方式（红线，见第八节）

* \*\*交付类型「外部链接」\*\*给视频素材类（#24 产品视频/教程库）：大体积媒体不进站（frp 隧道带宽是全站共享的稀缺资源），存网盘/素材中台链接

* 支持新增清单外的资料条目

### 5.4 资料详情与版本管理（/materials/:id）

页面主体是**版本时间线**（最新在上）：

```
● v3  2026-07-20 14:32  Kevin 上传   [下载] [预览]
│  修改说明：按顾问反馈补充了样品价
│  ✨ AI 差异概要：
│     · 新增：全色号样品包价格（$45/套）
│     · 修改：L1+ 客户折扣从 8% 调整为 8-12% 区间
│     · 删除：已停产的 613# 色号价格行
│
● v2  2026-07-18 09:15  亮哥 上传   [下载]
● v1  2026-07-17 16:40  亮哥 上传   [下载]
```

**版本规则**：

* 版本号条目内自增（v1, v2, v3…），上传即自动命名，用户不填版本号；数据库 `(material_id, version_no)` 唯一约束 + 冲突自动重试，两人同时上传不会撞号

* 下载文件名自动拼接：`价格体系_v3.xlsx`（由下载端点的 Content-Disposition 下发；原始文件名存档但不作为对外名称，避免"最终版-真最终版"混乱——这正是这个站要解决的问题）

* 上传新版本时可填一句「修改说明」（选填，与 AI 概要互补：一个是人的意图，一个是机器的核对）

* **单文件上限 50MB**：上传前端校验 + 后端强制。超限提示改用「外部链接」类型备注网盘地址——上传下载全程过 frp 隧道，大文件会占满全站带宽（展会模块踩过这个坑）

* **删除**：单版本删除 + 整条资料删除均为软删除（数据库标记 + 文件保留），审计留痕，**删除后下载端点立即拒绝访问**；不做回收站界面，误删找亮哥数据库恢复

* **「上一版」的统一口径**：当前版本 = 未删除的最大版本号；AI 差异对比的"上一版"同口径。删掉 v3 后当前版本回落 v2，下一次上传成为 v4（版本号只增不复用），其差异对比对象是 v2

* 文件类型不限制；预览支持图片 / PDF / MD（渲染前必须 sanitize，见第八节），其他类型仅下载

### 5.5 AI 差异概要管线

设计原则：**本地先算精确 diff，AI 只负责把 diff 讲成人话**。不把两份全文塞给模型——全量客户数据、产品目录这类最重要的材料恰恰最大，全文对比要么超上下文要么算错数字（价格体系上的幻觉差异比没有概要更危险）。

```
新版本落盘 → 抽取本版 & 上一版文本
→ 本地 diff（文本类：difflib unified diff；xlsx：openpyxl 行/单元格级结构对比）
→ diff 结果喂给平台 ai.service（专用 preset）生成概要 → 存入版本记录 → 前端轮询可见
```

* **文本抽取**：md / txt 直读；docx（python-docx）；pdf 用 pypdf 的 `extract_text()`（售后 SOP 模块已有同款先例，依赖已在 requirements）——**扫描版 PDF 抽出为空时落「不支持对比」，不拿空文本硬比**；xlsx 用 openpyxl（`data_only=True` 取公式缓存值，全部 sheet 逐个对比）

* **概要格式约定**（写进 preset 提示词）：按「新增 / 修改 / 删除」分类的 3\~8 条要点 + 一句话总评；仅陈述差异，不评价内容好坏

* **diff 超长**：截断至安全长度并在概要头部标注「变更过多，仅摘要前 N 处」

* **降级策略**：v1 无上一版 → 显示「首个版本」；抽取失败或图片/压缩包等 → 「该类型不支持对比」；AI 调用失败 → 标记失败可手动重试，**不影响版本本身的保存**

* 状态字段：`pending / done / failed / not_applicable`；**服务重启会丢在途后台任务，启动时扫描回收超时 pending**（展会模块 STALE\_PENDING 看门狗同款做法），杜绝永远转圈的"生成中"

### 5.6 全站动态（/activity）

审计日志的用户侧呈现：时间倒序流水，按人 / 按对象类型筛选。满足「以此用户名记录修改人员和修改时间」的核验需求——出了问题能追，平时当项目日报看。

### 5.7 预置初始化

迁移 + seed 脚本一次性写入：

* 1 条项目记录（数据模型带 project 维度，本期只有 1 个项目，为未来其他咨询项目复用留口，成本几乎为零）

* 35 项材料条目：从《企业需准备材料清单》解析编号、名称、说明、重要级、服务节点；**Phase 归属严格按清单「准备顺序」一节的四批映射**（Phase 1 = 1-5, 10-11, 14-18…），不按重要级推导——清单里 #29 是🟡却属 Phase 2、#12/13 是🔴却不在 Phase 1，两个维度不是一回事

* 交付类型标记：#14-17、#21、#31 凭据类 → 线下交付；#24 视频素材 → 外部链接

* 初始任务：材料清单「关键缺口」的 5 项 workshop（客户等级标准 / 三店归属规则 / 转化差归因 / 售后 SOP / 客户类型定义）各建一条任务

* 用户名白名单

## 六、Phase 2 功能设计（第一阶段上线后启动）

### 6.1 MD 在线编辑

* MD 类型的版本详情提供「基于此版本编辑」：分屏编辑器（左源码右预览）

* **保存即另存新版本**（自动 v+1，永不覆盖原版本），AI 差异管线照常触发——编辑和上传走完全相同的版本通道，模型上没有特例

* 并发保护：打开编辑器时记录基线版本号，保存时若已有更新版本则提示「已有人先保存了 vN，你的修改将基于旧版生成新版本」，让用户自行决定（人少，不做实时协同）

### 6.2 划线评论（参考飞书文档协作逻辑）

* **文本锚点评论**：MD 渲染视图中选中文本 → 浮出「评论」按钮 → 右侧评论栏定位显示；被评论文本高亮下划线

* 锚点存储：版本 ID + 引用原文 + 前后文特征（版本更新后在新版本中按文本匹配尝试重新锚定，匹配不到则降级为「挂在 vN 的评论」，不丢失）

* **文件级评论**：非 MD 文件在版本下方通用评论区

* ⚠️ **范围收窄需确认**（见第十节）：飞书对 PDF/图片预览也支持框选评论，本设计的划线评论仅覆盖 MD，PDF/xlsx 降级为文件级评论——顾问的修改意见恰恰可能长在 xlsx/PDF 上，这个取舍要亮哥拍板

* 评论支持回复（单层）、**解决/未解决状态**——顾问提的修改意见被处理后标记解决，未解决数量在资料列表露出角标

* 评论人即当前用户名，记审计

## 七、技术架构

### 7.1 后端（方舟平台内新领域模块）

* `app/pm/` 四件套：router / models / schemas / service（service 内拆 material\_service / diff\_service）

* **鉴权**：不接平台 RBAC。`POST /api/pm/entry` 白名单换 token；其余端点统一走自定义依赖 `require_pm_member`（验签 + 回查白名单 + 解析用户名供审计）。**check\_conventions 豁免方式：把** **`require_pm_member`** **登记进脚本的 AUTH\_PATTERNS**，而不是把 router 文件加白名单——这样每个端点仍被机器强制要求带鉴权依赖，将来漏写查得出来

* **数据表**（Alembic 迁移，命名遵守平台宪法 `ark_<domain>_<entity>` 复数 + 标准列名）：

| 表                                        | 用途                                                          |
| ---------------------------------------- | ----------------------------------------------------------- |
| `ark_pm_projects`                        | 项目（本期 1 条）                                                  |
| `ark_pm_members`                         | 用户名白名单                                                      |
| `ark_pm_tasks` / `ark_pm_task_materials` | 任务 / 任务-资料关联                                                |
| `ark_pm_materials`                       | 资料条目（名称项目内唯一，软删标记）                                          |
| `ark_pm_material_versions`               | 版本（文件路径、版本号、修改说明、AI 差异概要及状态；`(material_id, version_no)` 唯一） |
| `ark_pm_comments`                        | 评论（Phase 2，含锚点字段）                                           |
| `ark_pm_activity_logs`                   | 审计日志                                                        |

* **文件存储（安全关键）**：存 `REPO_ROOT/backend/data/pm/{material_id}/{uuid}{ext}`——**绝不放** **`backend/uploads/`**：该目录被 StaticFiles 无鉴权公开挂载在主域名 `/uploads` 下，商业数据放进去等于公网裸奔，且任意成员传 `.html` 会变成主站存储型 XSS。需确认服务器备份脚本覆盖新目录（材料唯一副本不能只在单机磁盘上）

* **下载/预览一律走鉴权端点**，不走静态：浏览器 `<a href>` 下载和 PDF 内嵌预览不带 Authorization header（素材模块踩过、有现成解法）——照抄素材模块的**短时效签名 URL**模式；端点内校验软删状态、下发 `Content-Disposition`（顺带实现自动重命名）+ `X-Content-Type-Options: nosniff`，HTML 类强制 attachment 不内联渲染

* **AI**：只走 `app.ai.service.chat`，新建后台可配 preset `pm_diff`。diff 任务用 FastAPI BackgroundTask（单机低频够用，不上任务队列）；**`chat()`** **需要活 session，而请求级** **`get_db`** **在后台任务执行前已关闭——后台任务内自建** **`SessionLocal()`** **+ finally 关闭**，归入平台红线 4 的线程池许可场景并注释说明

* 单条版本的 AI 失败不影响上传主流程（异常捕获 + logger + print，遵守平台红线 6）

* **测试范围**：版本状态机（软删后当前版本回落/版本号不复用）、token 签发/过期/移名单即失效、并发上传版本号唯一——这三处是本站的"钱和货"，按平台同等标准写 pytest

### 7.2 前端（完全独立应用）

* 新建顶层目录 `frontend-pm/`：独立 Vite + Vue 3 工程，与 `frontend/` 互不引用

* **不用 Element Plus**：页面少（6 个）、组件简单（列表/看板/上传/时间线），自研样式 + 少量无头逻辑，换来完全独立的视觉气质；有自己的 `tokens.css`，与方舟 DESIGN.md 无关。代价已计入排期（见第十节，质感与工期二选一已明示）

* 独立的轻量 API 层（不共用方舟 `api/clients.js`——两个站点、两套鉴权，共用反而耦合）

* **MD 渲染必须 sanitize**（marked/markdown-it + DOMPurify）：MD 里的原生 HTML 不消毒就是存储型 XSS，偷的是全员 token

* 单文件 >500 行拆 composable 的纪律沿用；⚠️ `check_conventions.py` 的前端检查只扫 `frontend/src/`，对 `frontend-pm/` 是零覆盖——实施时给脚本加上 `frontend-pm/src/` 前缀，不靠自觉

### 7.3 部署

## 八、安全（必须直说的风险）

材料清单里有**全量客户数据（#18）、价格体系（#2）、近 12 个月成交数据（#19）**——这是公司核心商业数据，而站点验证是无密码用户名。风险要认，缓解要做：

1. **文件不落公开静态目录**（§7.1）：这是比入口验证更要命的一层。文件全部走鉴权端点 + 短时效签名 URL，软删即不可下载
2. **凭据类材料（#14-17、#21、#31）禁止上传原文**——条目标记「线下交付」，上传入口置灰并说明原因，站上只跟踪状态
3. 用户名白名单**不用真名拼音**，用带后缀的变体（如 `liang.xz26`），口头分发
4. 站点全局 `noindex` + robots.txt 拒绝收录；不出现在任何对外链接
5. 验证接口限速（按用户名维度；IP 维度待确认 X-Forwarded-For 链路后启用，见 §3.1）；失败提示不区分原因
6. token 每请求回查白名单（移除单人立即生效）+ 全局版本号 salt 兜底（极端情况一键全员重新验证）
7. **MD 渲染与评论内容 sanitize**（§7.2），杜绝存储型 XSS
8. 可选加固（默认不做，亮哥拍板）：Nginx 层 IP 白名单限公司出口 IP——代价是出差/居家无法访问

**明确不防的**：内部成员之间互相冒用用户名（无密码机制天然防不了）。靠顶栏身份常显 + 审计留痕 + 小团队信任兜底，这是「简单进入」需求的固有代价。
