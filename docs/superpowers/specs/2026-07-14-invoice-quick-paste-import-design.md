# 发票 Excel 快速粘贴导入设计

日期：2026-07-14
状态：已完成业务方案确认，待实现

## 1. 问题与目标

历史流程需要先整理 Excel、另存为导入文件、上传到 OKKI、手工映射字段，再回到订单中检查结果。核心浪费不是“缺少上传按钮”，而是同一批明细被重复搬运和映射。

本功能的目标是：业务员在方舟新建发票时，从 Excel 复制产品明细，直接粘贴、校验并加入当前发票，减少重复录入，同时不绕过现有发票保存、校验和 OKKI 同步流程。

验收目标：

- 标准六列数据可以在 60 秒内完成粘贴、校验和加入发票。
- 唯一匹配的产品无需逐行操作。
- 无匹配或多匹配产品不能静默导入。
- Excel 成交价不会被系统客户价静默覆盖。
- 预览阶段不写数据库，不产生发票或定制产品。

## 2. 范围与非目标

### 2.1 本期范围

- 入口位于当前“新建/编辑发票”的明细区域，和“新增明细”并列。
- 用户必须先选择客户、订单类型和币种。
- 支持从桌面版 Excel/WPS 复制单元格后粘贴。
- 支持标准六列：Product、Length、Color、Weight、Quantity、Unit Price。
- 支持中英文表头别名、无表头固定六列、空白行和多余列。
- 批量执行产品匹配、库存 SKU 关联和客户价格查询。
- 用户处理阻断项后，将结果追加到当前发票编辑状态。

### 2.2 非目标

- 不新增独立导航页面。
- 不在第一版支持上传 Excel 文件。
- 不在导入完成时自动保存发票或同步 OKKI。
- 不做基于相似度的自动产品选择。
- 不把桌面资料包中的 Excel 文件作为生产运行依赖。
- 不新增数据库表或导入历史表。

## 3. 入口与用户流程

### 3.1 前置状态

“从 Excel 粘贴”按钮只在以下条件满足时可用：

- 已选择客户；
- 已选择库存单或生产单；
- 已选择币种。

条件不满足时按钮保持可见但禁用，并直接说明缺少的条件。

### 3.2 三步流程

1. 粘贴明细
   - 打开导入抽屉或对话框。
   - 显示当前客户、订单类型和币种，避免用户忘记上下文。
   - 用户直接把 Excel 单元格粘贴到输入区。
   - 不调用浏览器剪贴板权限接口，只处理用户主动触发的 paste 事件。
2. 校验并修正
   - 前端完成表格结构解析。
   - 后端一次性执行规范化、产品匹配和价格解析。
   - 页面展示通过、提醒、阻断三类结果。
   - 用户逐行解决无匹配或多匹配问题。
3. 加入当前发票
   - 所有阻断项解决后才允许继续。
   - 明细追加到现有编辑表格，保持原始顺序。
   - 返回发票编辑器，继续填写联系人、运费、付款条款和备注。

“加入当前发票”不是保存动作，结果仍可逐行编辑和删除。

## 4. 粘贴解析规则

### 4.1 表头识别

优先识别第一条非空行为表头。标准字段及初始别名：

| 标准字段 | 可识别表头示例 |
| --- | --- |
| product | Product、产品、产品名称、产品名称 Product |
| length | Length、长度 |
| color | Color、Colour、颜色、色号 |
| weight | Weight、重量、克重 |
| quantity | Quantity、Qty、数量 |
| unit_price | Unit Price、Price、单价、成交价 |

如果第一行不是可识别表头且数据恰好为六列，则按标准顺序识别，并在预览中明确显示识别映射。不能确认列结构时不猜测，要求用户重新复制包含表头的数据。

### 4.2 值规范化

| 字段 | 规则 |
| --- | --- |
| Product | 去除首尾及重复空格；使用产品中英文别名归一化后参与匹配 |
| Length | 兼容 `18`、`18 inch`、`18\"`，统一为系统长度字符串 |
| Color | 去除首尾空格，统一大小写和分隔符，按色号别名归一化 |
| Weight | 兼容 `100g`、`100 g`、`0.1kg`，统一换算为整数克 |
| Quantity | 必须是大于 0 的整数 |
| Unit Price | 必须大于 0，按当前币种保留两位小数 |

空白行自动忽略；保留原 Excel 行号。多余列忽略但给出提示；缺少必填列时停止进入产品匹配。第一版单次最多处理 200 行，超过时提示用户拆分导入。

不自动合并重复行。这样可以保持来源数据和成交价差异，避免系统替用户改变订单含义。

## 5. 产品匹配规则

产品匹配必须是确定性的。标准化后按以下维度匹配：

- 产品系列或 `product_display`；
- 长度；
- 颜色；
- 克重或销售单位。

处理结果：

- 唯一匹配：返回 product_id、sku_id、产品名称和库存信息。
- 多个匹配：返回候选项，阻断该行，要求人工选择。
- 无匹配的库存单：阻断，必须选择已有产品。
- 无匹配的生产单：默认阻断；用户可明确选择“作为定制产品”，最终保存时沿用现有定制产品创建流程。

禁止按相似度自动选择最接近产品。候选项可以按精确维度和名称相关度排序，但系统不得代替用户确认。

历史资料中的产品中英文对照和颜色表只用于建立规范化别名及回归测试样本。生产查询仍以方舟当前产品数据为准。

## 6. 价格规则

- Excel Unit Price 是本次成交价，预览和加入发票时保持不变。
- 后端同时解析当前客户价格，并返回系统价和差额。
- Excel 价与系统价不同：状态为提醒，不阻断；价格来源记为 manual。
- Excel 价与系统客户价相同：沿用 customer_rule 价格来源。
- 系统没有可用价格但 Excel 有合法价格：保留 Excel 价，标记 manual 并提醒。
- 页面同时展示 Excel 成交价、系统客户价和差额，不能只显示警告文案。
- 最终保存仍由后端按现有逻辑重新计算金额，禁止信任前端合计。

## 7. 校验状态与错误反馈

单行状态按以下优先级计算：

1. 字段错误；
2. 产品匹配错误；
3. 价格差异提醒；
4. 通过。

顶部汇总显示总行数、通过数、提醒数和阻断数。用户点击汇总数字可定位到对应行，但不隐藏其他行，避免失去整体上下文。

错误文案必须指向行动，例如：

- “第 8 行缺少 Weight，请回到 Excel 补充后重新粘贴。”
- “第 12 行找到 2 个产品，请选择正确的 SKU。”
- “成交价比客户价低 2 USD；将保留 Excel 成交价。”

网络错误保留当前粘贴内容和本地解析结果，用户可以直接重试，不需要重新复制。

## 8. 技术方案

采用“前端解析结构 + 后端批量预检”的混合方案。

### 8.1 前端职责

- 读取 paste 事件中的 `text/plain` 制表符数据；
- 解析表头、列和原始行号；
- 在请求前发现缺列、列数不一致和基础数值错误；
- 展示批量预检结果及候选产品；
- 将确认结果转换成现有 InvoiceItem 编辑模型；
- 防止同一批数据在当前编辑会话中重复追加。

### 8.2 后端职责

- 重新校验所有字段，不信任前端解析结果；
- 统一产品名称、颜色、长度和克重；
- 批量查询产品、SKU 和库存；
- 批量解析客户价格；
- 返回行级状态、候选项和规范化值；
- 不在预检接口中执行任何数据库写入。

### 8.3 API

新增：

```text
POST /api/invoice/import/preview
Permission: invoice:write
```

请求结构：

```json
{
  "customer_id": 123,
  "order_type": "stock",
  "currency": "USD",
  "rows": [
    {
      "source_row": 2,
      "product": "Standard Double Drawn Genius Weft",
      "length": "18",
      "color": "#1B",
      "weight": "100g",
      "quantity": 2,
      "unit_price": "36.00"
    }
  ]
}
```

响应继续使用统一信封，`data` 中包含：

```json
{
  "batch_fingerprint": "...",
  "summary": {
    "total": 1,
    "passed": 1,
    "warning": 0,
    "blocked": 0
  },
  "rows": [
    {
      "source_row": 2,
      "normalized": {},
      "matched_product": {},
      "candidates": [],
      "customer_price": "36.00",
      "price_difference": "0.00",
      "status": "passed",
      "errors": [],
      "warnings": []
    }
  ]
}
```

候选产品只返回完成决策所需字段，不返回无关业务数据。候选列表设置合理上限；超出时提示用户缩小条件或使用现有产品选择器。

### 8.4 重复导入防护

- 后端根据客户、订单类型、币种和规范化行数据生成 `batch_fingerprint`。
- 前端在当前发票编辑会话保存已追加的指纹。
- 再次追加相同指纹时阻止操作，并提示“这批数据已经加入当前发票”。
- 点击“加入当前发票”后立即禁用按钮，直到追加动作结束。
- 最终保存接口继续使用现有发票更新语义，不额外创建导入记录。

## 9. 代码落点

预计最小改动范围：

- `backend/app/invoice/router.py`：注册批量预检端点和权限。
- `backend/app/invoice/schemas.py`：新增预检请求与响应模型。
- `backend/app/invoice/import_service.py`：解析规范化、批量匹配和价格差异编排。
- `backend/tests/`：新增导入预检服务和 API 测试。
- `frontend/src/api/invoice.js`：登记批量预检调用。
- `frontend/src/views/invoice/components/InvoicePasteImport.vue`：三步导入界面。
- `frontend/src/views/invoice/composables/useInvoicePasteImport.js`：粘贴解析、状态和重复防护。
- `frontend/src/views/invoice/InvoiceManage.vue`：入口和导入结果回填。
- `docs/api-reference.md`：登记新端点。

主页面保持薄壳，避免继续扩大发票编辑器单文件。

## 10. 测试与验收

### 10.1 前端测试

- 中文、英文和混合表头解析；
- 无表头固定六列识别；
- 空白行、多余列、缺失列和列数错位；
- 长度、颜色、克重、数量和价格规范化；
- 阻断项未解决时导入按钮不可用；
- 候选产品选择后解除阻断；
- 价格差异保留 Excel 价格；
- 相同批次不能重复追加；
- 200 行边界和超限提示。

### 10.2 后端测试

- 唯一匹配、多匹配和无匹配；
- 库存单无匹配必须阻断；
- 生产单明确选择后允许定制路径；
- 客户价相同、不同和缺失；
- 非法数量、价格、克重和币种；
- 权限校验和统一响应信封；
- 预检过程不产生数据库写入；
- 批量查询没有逐行 N+1 查询退化。

### 10.3 真实场景验收

- 从资料包的标准工作表复制六行并成功导入；
- 从历史发票模板只复制产品表区域并成功识别；
- 制造一个重复产品候选，确认必须人工选择；
- 修改一行 Unit Price，确认价差可见且成交价保持不变；
- 连续点击导入，确认明细不会重复；
- 导入后保存草稿、重新打开、验证发票和同步 OKKI 的现有流程不回归。

完成时执行：

```bash
cd backend && pytest
cd frontend && npm run build
python scripts/check_conventions.py
```

## 11. 发布策略

- 第一版仅开放给已有 `invoice:write` 权限的用户，不新增权限码。
- 先在开发环境用真实历史 Excel 样本验收，再部署生产。
- 上线后观察阻断原因分布；如果大量集中在产品别名或色号别名，再考虑建设可维护的别名管理，而不是第一版提前新增配置系统。
- 不自动迁移或回写历史发票。
